# 04 — التوثيق البرمجي لتطبيق الموبايل (Flutter)

مشروع Flutter واحد يولّد تطبيقَي **أندرويد** و**آيفون**، ويتصل بنفس سيرفر النسخة الويب عبر الـ API. المشروع في مجلد `mobile_flutter/`.

---

## 1. بنية المشروع

```
mobile_flutter/
  pubspec.yaml          ← اسم التطبيق، الإصدار، المكتبات
  lib/
    main.dart           ← نقطة البداية + التطبيق + الدخول + الهيكل/التنقّل
    api.dart            ← الاتصال بالـ API + النماذج + حفظ الجلسة
    theme.dart          ← الألوان، الثيم، شارة الحالة، أدوات الفئات
    imam.dart           ← شاشات قيّم المسجد + بطاقة الغرض + رفع غرض
    ministry.dart       ← طابور موافقات الوزارة + بطاقة العملية + السجل
    manager.dart        ← لوحة المدير + شاشة المساجد
```

## 2. المكتبات (pubspec.yaml)

| المكتبة | الغرض |
|--------|------|
| `http` | إرسال طلبات الـ API |
| `shared_preferences` | حفظ رمز الدخول وبيانات المستخدم على الجهاز |
| `image_picker` | التقاط صورة الغرض من الكاميرا/المعرض |

## 3. شرح الملفات

### `api.dart` — طبقة الاتصال

- **`kApiBase`**: عنوان السيرفر. غيّره حسب البيئة (محاكي أندرويد `10.0.2.2`، أو نطاق المديرية عند النشر).
- **`AppUser`**, **`Item`**: نماذج تُبنى من JSON. `Item.photoUrl` يحوّل المسار النسبي إلى رابط كامل.
- **`Api`** (ثابتة): تحفظ `token` و `user`، وتوفّر:
  - `loadSession()` / `logout()` — تحميل/مسح الجلسة من التخزين.
  - `requestOtp` / `verifyOtp` — المصادقة.
  - `items(scope)` / `addItem` / `deleteItem` / `requestItem` / `deliver` — الأغراض.
  - `requests` / `approve` / `reject` — الموافقات.
  - `stats()` — الإحصاءات.
  - الدالة الداخلية `_req` توحّد إرسال الطلبات وإضافة ترويسة المصادقة ومعالجة الأخطاء.

### `theme.dart` — الهوية

- `AppColors`: لوحة الألوان (أخضر/ذهبي) المطابقة للويب.
- `buildTheme()`: ثيم Material 3.
- `StatusBadge`: شارة ملوّنة لحالة الغرض.
- `categoryEmoji` و `categories`: فئات الأغراض ورموزها.

### `main.dart` — التطبيق والتنقّل

- `main()`: يحمّل الجلسة المحفوظة ثم يشغّل `AwqafApp`.
- `AwqafApp`: يضبط `MaterialApp` ويفرض اتجاه **RTL** على كامل التطبيق، ويعرض شاشة الدخول أو الهيكل حسب وجود جلسة.
- `LoginScreen`: إدخال رقم الجوال → طلب OTP → إدخال الرمز → دخول. يعرض الرمز التجريبي تلقائياً.
- `HomeShell`: يبني شريط تنقّل سفلياً **حسب الدور** ويعرض الشاشة المناسبة. يُظهر زر «+» لرفع غرض للقيّم في تبويب «أغراضي».

### `imam.dart` — شاشات القيّم

- `ItemCard`: بطاقة غرض (صورة، حالة، تفاصيل، زر إجراء).
- `BrowseScreen`: تصفّح أغراض المساجد الأخرى مع زر «طلب».
- `MyItemsScreen`: أغراض مسجدي مع إجراءات (سحب الإعلان / تأكيد التسليم).
- `MyRequestsScreen`: متابعة طلباتي وحالاتها.
- `AddItemScreen`: نموذج رفع غرض مع **التقاط صورة** عبر `image_picker` وتحويلها إلى Base64 وإرسالها.

### `ministry.dart` — شاشات الوزارة

- `OpCard`: بطاقة عملية (مصدر ← طالب).
- `MinistryQueueScreen`: طابور الموافقات مع زرّي **موافقة/رفض**؛ الرفض يفتح ورقة سفلية لإدخال السبب.
- `OperationsLogScreen`: سجل كل العمليات (يُستخدم للوزارة والمدير).

### `manager.dart` — شاشات المدير

- `ManagerDashboard`: بطاقات إحصائية + مخطّط الفئات تداولاً.
- `MosquesScreen`: قائمة المساجد المشاركة مع عدد ما رفعته وطلبته.

## 4. البناء والتشغيل

```bash
cd mobile_flutter
flutter pub get
flutter create .          # لإنشاء مجلدي android/ و ios/ إن لم يوجدا
flutter run               # تشغيل على جهاز/محاكي
```

### أذونات الكاميرا
- أندرويد: أضِف `CAMERA` في `AndroidManifest.xml`.
- آيفون: أضِف `NSCameraUsageDescription` في `Info.plist`.

(التفاصيل في `mobile_flutter/README.md`.)

### الإصدارات النهائية
```bash
flutter build appbundle --release   # أندرويد → Google Play
flutter build ios --release         # آيفون (على Mac) → App Store عبر Xcode
```

## 5. ملاحظات

- التطبيق **عديم الحالة على الخادم**: يعتمد كلياً على الـ API، فأي تحديث في منطق العمل يكفي تطبيقه في الخادم.
- منطق الصلاحيات الحقيقي مفروض في الخادم؛ التطبيق يعرض الواجهات المناسبة للدور فقط لتجربة أفضل.
- لإضافة الإشعارات الفورية لاحقاً: تُدمج `firebase_messaging` ويُضاف مسار تسجيل الجهاز في الخادم.
